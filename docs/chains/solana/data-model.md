# Data Model

Solana's data shapes are unfamiliar to EVM-trained indexers. This file enumerates the on-the-wire types you'll see.

## Block

Per slot (when the leader produces one):

```
{
  "blockHeight": 245678901,            // monotonic block index
  "blockTime": 1715000000,             // unix timestamp (leader-set)
  "blockhash": "B58‚Ä¶",                 // 32-byte slot hash, base58
  "parentSlot": 245678900,
  "previousBlockhash": "B58‚Ä¶",
  "transactions": [...],
  "rewards": [...]                     // staking and tx-fee rewards
}
```

The block is fetched by **slot number**, not block height. `getBlock(245678901)` returns the block produced at that slot ‚Äî or `null` if the slot was skipped.

## Transactions

```
{
  "transaction": {
    "signatures": ["B58‚Ä¶", ...],
    "message": {
      "accountKeys": ["B58‚Ä¶", ...],     // pubkeys
      "instructions": [
        {
          "programIdIndex": 7,           // index into accountKeys
          "accounts": [0, 3, 5],         // indexes into accountKeys
          "data": "base58_or_base64"     // serialized instruction args
        }
      ],
      "recentBlockhash": "B58‚Ä¶",
      "header": { ... }
    },
    "version": "legacy" | 0,
    "addressTableLookups": [             // V0 only
      {
        "accountKey": "B58‚Ä¶",            // ALT account
        "writableIndexes": [...],
        "readonlyIndexes": [...]
      }
    ]
  },
  "meta": {
    "err": null | { ... },
    "fee": 5000,                         // lamports
    "preBalances": [...], "postBalances": [...],
    "preTokenBalances": [...], "postTokenBalances": [...],
    "innerInstructions": [
      { "index": 0, "instructions": [...] }
    ],
    "logMessages": ["Program 11111‚Ä¶ invoke [1]", ...],
    "computeUnitsConsumed": 12345,
    "loadedAddresses": { "writable": [...], "readonly": [...] }    // V0
  }
}
```

Notable:
- **`fee`** is the base fee in lamports (10^-9 SOL). Priority fee is included in this if compute-budget instructions added one.
- **`preBalances` / `postBalances`** are the SOL balances of every account in `accountKeys` before / after the tx. **Use these to compute SOL transfers.**
- **`preTokenBalances` / `postTokenBalances`** are SPL Token balance changes (decoded from the SPL Token account state).
- **`logMessages`** is an array of strings. Includes program invocation markers, `msg!()` output, and program-specific binary logs (base64-encoded).
- **`innerInstructions`** is the CPI call tree (see [architecture.md](architecture.md)).
- **`loadedAddresses`** (V0) is the resolved ALT references ‚Äî pubkeys not directly in `accountKeys`.

## Instructions

The atomic unit of program invocation. Two views:

### Top-level (in `message.instructions`)
- Submitted by the user.
- Each names a program and a list of accounts.
- `data` is the program-specific argument blob (typically Borsh-serialized, sometimes Anchor-discriminator-prefixed).

### Inner (in `meta.innerInstructions`)
- Generated by program execution via CPI.
- Same shape as top-level, but appended by the runtime.

A complete "what happened" picture includes both. **Indexers building call trees must traverse innerInstructions in order**; they're emitted in execution order.

## Logs and event reconstruction

Solana has no first-class log/event mechanism. Three patterns indexers use:

### 1. Program-emitted text logs (`msg!()`)

Plain strings appended to `logMessages`. Free-form. Examples:
```
"Program log: Instruction: SwapBaseInput"
"Program log: Swap amount_in: 1000000, amount_out: 950000"
```

Parsing these is **brittle** ‚Äî they're effectively `printf` debug output. dApps that care about structured events use pattern (3) below.

### 2. Inner-instruction-as-event

The CPI to a particular program with particular arguments **is** the event. SPL Token's `Transfer` instruction acts like a `Transfer` event. Indexers reconstruct token transfer history by scanning **inner instructions where `programId == SPL Token`** and decoding their args.

This is the most reliable pattern for built-in programs.

### 3. Anchor-style program logs (`emit_cpi!`, sol_log_data)

Programs built with Anchor (or hand-rolled) can emit binary-encoded events as a CPI to a specific instruction or via the `sol_log_data` syscall. These appear in `logMessages` as base64-encoded blobs:

```
"Program data: <base64>"
```

Indexers decode the base64, then Borsh-deserialize against the program's IDL (interface definition).

For Anchor programs, the IDL is published; for non-Anchor programs, decoding requires reverse-engineering or program-specific knowledge.

## Account state diffs

The complement to logs. For any tx, you can compare `preBalances` / `postBalances` and the post-tx account state (via `getAccountInfo` at the resulting slot) to "see what changed."

For SPL Token specifically, `preTokenBalances` / `postTokenBalances` give per-tx balance deltas decoded into token amounts. **This is the canonical way to track SPL token transfers** ‚Äî log scanning is supplementary.

## Transactions vs signatures

- A transaction has potentially many `signatures` (multiple signers).
- The **first signature is conventionally the "tx hash"** ‚Äî the canonical identifier used for retrieval (`getTransaction(signature)`).

JSON-RPC accepts the signature as a base58 string. Indexers usually store both binary and base58 forms for flexibility.

## V0 vs Legacy transactions

| Version | Notable |
|---|---|
| `legacy` | Original; all accounts in `accountKeys` directly |
| `0` (V0) | Address Lookup Tables (ALTs); supports more accounts per tx |

V0 was introduced to support large MEV / aggregator txs (Jupiter, etc.) that need to reference dozens of accounts.

## Compute units and fees

Solana fees:
- **Base fee**: 5000 lamports per signature ({{unsourced: confirm; can be configured upward}}). Always paid.
- **Priority fee** (optional): set via `ComputeBudget.SetComputeUnitPrice` in microlamports per CU. Multiplied by `compute_units_consumed`.

Total fee in lamports = base fee √ó signatures + (CU price √ó CUs consumed) / 1e6.

Indexers tracking "total fees paid" per slot must aggregate from receipts.

## State

There is no Merkle Patricia Trie. State is the union of all account states (raw bytes per account). Account states are versioned at each slot ‚Äî historical state is **not retained by default** in the validator.

For "state at past slot N", you need:
- An archival validator (extremely rare; few exist).
- A snapshot service (Helius, Triton, etc.).
- The state-diff history reconstructed from `preBalances`/`postBalances` + account writes.

This is **fundamentally different** from EVM where archive nodes give random historical access. Plan accordingly.
